# Issue Design Patterns

## 이슈 분할 원칙
- 하나의 이슈는 하나의 관심사(concern)만 다룬다
- 이슈 크기: 1~3일 작업량이 적절 (너무 크면 분할)
- 의존성이 있는 이슈는 Phase로 구분하고 순서를 명시한다
- 각 이슈는 독립적으로 테스트 가능해야 한다

## 좋은 이슈의 요건
- 배경(Why): 왜 이 작업이 필요한지 맥락 설명
- 목표(What): 달성해야 할 구체적 결과물
- 설계(How): 구현 방법의 기술적 설계
- 구현 계획: Phase별 체크리스트로 진행 추적 가능
- 파일 변경 요약: 어떤 파일을 만들고/수정하는지 테이블
- 리스크: 예상 문제와 대응 방안
- 완료 기준: 이슈가 "완료"인지 판단할 수 있는 체크리스트

## 이슈 분할 판단 기준
- 서로 다른 모듈/레이어를 수정하면 분할
- 독립적으로 배포/테스트 가능하면 분할
- 한 이슈에서 파일 변경이 15개 이상이면 분할 고려
- Phase 간 의존성이 약하면 별도 이슈로

## 이슈 간 의존성 표현
- 의존하는 이슈: "Depends on #N" 또는 "Blocked by #N"
- 관련 이슈: "Related to #N"
- 순서: Phase 1 → Phase 2 형태로 구현 순서 명시

## 라벨 전략
- `enhancement`: 새 기능
- `bug`: 버그 수정
- `refactor`: 리팩토링
- `docs`: 문서
- `breaking`: 호환성 깨지는 변경
